# Dev 3：Frontend / Product Engineer 开发计划

## 职责定位

Dev 3 负责 AgentOS MVP 的 Web 工作台和核心交互体验，包括组织雷达、项目状态、个人简报、共事界面、审批队列、审计日志和记忆管理。

这个角色要把权限边界、Agent 可解释性和工作流闭环清楚地呈现在界面里，让用户觉得系统可信、可用、不会像监控工具。

## 第一步：设计前端信息架构

### 要做什么

- 定义页面结构：Dashboard、Projects、Personal Brief、CoWork、Approvals、Knowledge、Audit、Admin。
- 定义不同角色的首页内容。
- 定义 Agent 回答组件：模式、来源、审批状态、下一步动作。
- 和 Dev 1 对齐 API mock。
- 和 Dev 2 对齐 Agent response schema。
- 和 QA 对齐核心页面验收标准。

### 交付什么

- 页面结构图。
- 前端路由设计。
- 核心组件清单。
- 角色化首页草图。

### 如何验收

- 管理层、团队负责人、员工的主路径清楚。
- 所有 Agent 输出都有统一展示方式。
- 私人数据、团队数据、管理数据在 UI 上有明确边界。
- QA 可以基于页面结构写 E2E 测试。

## 第二步：搭建 Web 工作台

### 要做什么

- 搭建前端工程。
- 实现登录后 app shell。
- 实现导航、页面布局、基础设计系统。
- 实现角色切换 mock。
- 接入 mock API 展示基础数据。

### 交付什么

- Web app shell。
- 基础导航。
- 角色化首页框架。
- Mock API adapter。

### 如何验收

- 用户可以进入所有 MVP 页面。
- 页面能根据角色展示不同模块。
- 前端可以在无真实后端时基于 mock 数据运行。
- 早期 `index.html` 草稿不进入正式前端工程范围。

## 第三步：实现组织雷达、项目状态、个人简报

### 要做什么

- 实现组织雷达页面：目标进展、项目风险、客户升级、团队负载、待管理层介入事项。
- 实现项目状态页面：进展、阻塞、依赖、待决策事项、行动项。
- 实现个人简报页面：今日优先级、会议准备、待回复事项、被阻塞任务、阻塞他人的任务。
- 展示 Agent 使用的数据来源和生成依据。

### 交付什么

- Dashboard 页面。
- Projects 页面。
- Personal Brief 页面。
- Source reference 展示组件。

### 如何验收

- Executive 能看到组织级聚合状态。
- Manager 能看到团队项目状态。
- Employee 能看到个人工作简报。
- 每条摘要都能显示来源或生成依据。
- 管理视图不呈现为员工监控界面。

## 第四步：实现审批和审计界面

### 要做什么

- 实现审批队列页面。
- 展示动作内容、目标工具、来源、影响范围、风险等级。
- 支持批准、拒绝。
- 展示审批状态变化。
- 实现审计日志页面。
- 审计日志支持按时间、用户、动作、风险等级筛选。

### 交付什么

- Approvals 页面。
- Audit 页面。
- Approval detail 组件。
- Audit filter 组件。

### 如何验收

- 用户能清楚判断一个动作是否应该批准。
- 审批状态变化能及时反映到 UI。
- 审计记录能追溯 Agent 做过什么、谁批准、结果是什么。
- 未授权用户看不到不属于自己的审批和审计记录。

## 第五步：实现共事模式和知识检索

### 要做什么

- 实现共事讨论界面。
- 明确区分私人讨论区和公开输出区。
- 支持从讨论生成 memo、消息草稿、任务草稿。
- 支持用户确认公开输出。
- 实现知识检索页面。
- 知识检索展示答案、事实/推断、出处。

### 交付什么

- CoWork 页面。
- Knowledge 页面。
- Public output confirmation UI。
- Fact / inference / source 展示组件。

### 如何验收

- 私人内容不会误展示成团队内容。
- 公开输出必须经过用户确认。
- 知识检索结果能查看出处。
- Agent 无来源回答在 UI 上明确标记不确定。
- 用户可以从共事讨论自然进入审批或发布流程。

## 第六步：试点版本体验收口

### 要做什么

- 完善加载状态、空状态、错误状态。
- 实现记忆管理基础页面。
- 优化窄屏和移动端适配。
- 根据 QA 和试点反馈修复 UI 问题。
- 补齐前端 README。

### 交付什么

- 试点版前端。
- Memory 页面基础版。
- UI 状态补齐。
- 前端 README。

### 如何验收

- 核心流程无需开发者解释即可使用。
- 管理层、负责人、员工看到不同首页和数据。
- 私人/团队/管理边界在 UI 上清楚。
- QA 的核心 E2E 测试通过。

## 与其他角色的协作接口

- 与 Dev 1：依赖 API contract、角色化数据、审批和审计接口。
- 与 Dev 2：依赖 Agent response schema、模式标记、来源引用、待审批动作。
- 与 Dev 4：依赖知识检索结果、source reference、敏感等级展示。
- 与 QA：提供页面稳定选择器、验收路径、测试账号角色。

## 最终完成标准

- Web 工作台完整支持组织简报、项目状态、个人简报、共事模式、知识检索、审批、审计。
- UI 能清楚表达 Agent 当前模式、数据来源、审批要求和权限边界。
- 用户能从 Agent 建议自然进入审批和审计闭环。
- 产品体验不让员工感觉被监控。
