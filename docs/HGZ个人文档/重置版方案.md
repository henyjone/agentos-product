# 管理日报与智能提交助手：系统重设计方案

## 一、现状与问题

当前仓库包含两个核心模块：

| 模块 | 现状 | 问题 |
|------|------|------|
| `commit_guide` | 交互式引导工具，按步骤让用户选择类型、输入范围与描述 | 流程繁琐，完全没有利用 diff 信息自动生成详细说明；与“自动生成详细提交说明，减少服务端 AI 消耗”的目标相悖 |
| `agent_engine` | 定义了 Agent 模式、路由、风险分类等抽象，但 chains 为空壳，orchestrator 仅返回骨架响应 | 没有实际的业务链，无法支撑日报生成与团队协调 |

用户期望获得两样东西：
1. **客户端工具**：在 `git commit` 前自动分析暂存区变更，生成结构化、详尽的提交说明，尽可能减少服务端重复 AI 分析。
2. **服务端工具**：定时从 Gitea 拉取全员提交记录，理解代码内容，生成个人与团队日报，识别风险并给出协调建议。

下文给出一个完整的新设计，可直接指导重构。

## 二、总体架构

```
┌───────────────────────────┐        ┌───────────────────────────────┐
│  开发者本地 (CLI)          │        │  服务器 (定时任务 / API)       │
│                           │        │                               │
│  git-smart-commit         │ push   │  daily-report-engine          │
│  ├─ diff 分析引擎         │───────▶│  ├─ Gitea 数据采集            │
│  ├─ 规则 / 本地小模型     │        │  ├─ agent_engine (链式处理)    │
│  ├─ 消息构造器            │        │  ├─ 日报生成 & 风险检测        │
│  └─ 提交 & 推送           │        │  └─ 通知 (企微/邮件)           │
└───────────────────────────┘        └───────────────────────────────┘
```

- **客户端**：工具名暂定为 `git-smart-commit`，重写现有 `commit_guide`。
- **服务端**：工具名暂定为 `daily-report-engine`，基于 `agent_engine` 扩展实现。

## 三、客户端：智能提交助手 `git-smart-commit`

### 3.1 设计目标

- 一行命令即可生成详细的提交说明，无需多步交互。
- 说明内容应包含：变更摘要、详细列表、影响模块、关联 issue（如有），减少服务端事后通过 AI 反向理解的消耗。
- 支持 `--auto` 模式直接提交，也支持预览编辑模式。
- 允许接入本地轻量模型做润色，但默认基于规则已能产出高质量信息。

### 3.2 核心流程

```
git add ...
git smart-commit [--auto] [--push]
     │
     ├─ 获取 git diff --cached 全文
     ├─ 分析变更：
     │    - 文件统计（新增/修改/删除）
     │    - 函数/类级变更（通过 AST 或正则）
     │    - 关键词识别（fix, add, remove, refactor 等）
     ├─ 推断 commit 类型（feat/fix/refactor/...）
     ├─ 识别影响范围（从文件路径推断模块名，如 commit_guide, agent_engine）
     ├─ 生成结构化消息：
     │    feat(agent_engine): 增加 diff 分析链
     │    
     │    - 新增 DiffAnalyzer 类，支持暂存区 diff 解析
     │    - 扩展 MessageBuilder 以生成多行正文
     │    - 修复解析中空文件导致的异常
     │    
     │    Related: #342
     ├─ 可选 AI 润色（本地小模型）
     └─ 提交 & 可选推送
```

### 3.3 消息格式规范

采用“标题 + 正文 + 脚注”的结构，便于服务端解析：

```
<type>(<scope>): <简短摘要>

<变更点列表>

<脚注>
```

- **标题**：遵循 Conventional Commits，`feat/fix/refactor/perf/docs/test/chore`，scope 从路径推断。
- **正文**：每条一个 `- ` 开头，描述具体变更内容与动机。
- **脚注**：`Related:` / `Closes:` 引用 issue 编号。

示例：
```
feat(commit_guide): 支持基于 diff 自动生成详细提交说明

- 新增 DiffParser，提取文件变更、函数增删
- 新增 CommitMessageBuilder，按模板生成多行消息
- 增加 --auto 和 --push 参数，支持一键提交流程
- 移除旧的交互式选择界面
```

### 3.4 技术方案

基于 Python，复用 `commit_guide` 的目录但完全重写逻辑：

| 组件 | 功能 |
|------|------|
| `diff_parser.py` | 解析 `git diff --cached` 输出，提取文件列表、函数变更等 |
| `type_inferrer.py` | 规则 + 关键词推断 commit 类型 |
| `message_builder.py` | 根据解析结果组装多行消息 |
| `cli.py` | Click/argparse 入口，整合流程 |
| `ai_enhance.py` | 可选，调用本地 Ollama 模型润色正文 |

去重交互：不再使用 `input()` 步步询问，仅在非 `--auto` 模式下用 `$EDITOR` 或简单确认。

### 3.5 与现有代码的关系

- 删除或归档 `commit_guide/main.py` 中的交互式 App 状态机。
- 保留 `types.py` 中的 `CommitType`、格式校验等，扩展消息格式。
- `git_utils.py` 可继续使用，增加 `get_diff`、`get_log` 等函数。
- `ai_assist.py` 重构为可选本地模型调用，而非必须依赖远程 API。

## 四、服务端：管理日报引擎 `daily-report-engine`

### 4.1 设计目标

- 每日凌晨（或按需）通过 Gitea API 拉取组织内所有仓库的提交、PR、Issue 数据。
- 利用 `agent_engine` 的链式处理生成：
  - 个人日报：每个成员的提交摘要、代码量、完成任务。
  - 团队日报：整体进度、跨模块协作情况、风险预警。
- 输出 Markdown 日报，可推送到企业微信、邮件或回写 Gitea Wiki。
- 持久化历史数据，用于趋势分析。

### 4.2 数据采集层

- **数据源**：Gitea REST API (v1)。
- **采集项**：
  - 组织下所有仓库列表。
  - 指定时间段内的 commits（按分支，默认 `main`/`master`）。
  - 每个 commit 的 diff（可选，用于详细理解，但成本高，可用客户端生成的消息替代）。
  - Pull Requests 及其状态、审查情况。
  - Issues（与 commit 关联的）。

### 4.3 智能分析层（基于 agent_engine）

重用 `agent_engine` 的抽象，实现具体 `AgentChain`：

| Chain | 功能 |
|-------|------|
| `PersonalBriefChain` | 根据某用户当日所有 commit message 和轻量 diff 摘要，生成个人工作摘要 |
| `ProjectStatusChain` | 汇总项目维度进展，识别阻塞（如长期未合并的 PR） |
| `RiskClassifier`（已有） | 检测高风险操作（直接推送保护分支、敏感文件修改等） |
| `TeamCoordinationChain` | 分析跨成员工作交集，推荐协作或提醒冲突 |

**内存模块 (`MemoryClient`)**：可存储员工历史工作特征、项目上下文，辅助生成更连贯的日报。

**模式路由 (`ModeRouter`)**：定时任务进入 `AgentMode.EXECUTION` 或新增 `AgentMode.DAILY_REPORT`。

### 4.4 日报模板示例

```markdown
## 团队日报 - 2025-05-02

### 总体概览
- 提交数：24
- 参与人：5
- 活跃仓库：3
- 风险项：2

### 个人简报
#### 张三 @zhangsan
- **提交数**：4
- **主要工作**：
  - feat(agent_engine): 增加 diff 分析链
  - fix(commit_guide): 修复解析空 diff 崩溃
- **代码量**：+230 / -45
- **关联任务**：#342, #355
- **风险**：无

#### 李四 @lisi
- ...

### 团队协作 & 风险
- **阻塞项**：PR #128 已等待审查 3 天，建议 @wangwu 尽快 review。
- **重复工作**：agent_engine 的 mode 定义与 commit_guide 的类型定义相似，建议统一。
```

### 4.5 实现方案

- 使用 `APScheduler` 或系统 cron 触发。
- 扩展 `agent_engine/chains/` 下的空文件，实现上述链。
- 新增 `gitea_client.py` 封装 API 调用。
- 新增 `report_renderer.py` 生成各类输出格式。
- 配置驱动：通过 `config.json` 指定 Gitea 地址、Token、日报接收渠道等。

## 五、整合与实施路线

### 阶段一：客户端工具（2-3 天）

- 重写 `commit_guide` 为 `git-smart-commit`。
- 输出第一个可用版本，支持 `--auto` 直接提交。
- 不依赖服务端，独立使用。

### 阶段二：服务端日报基础版（1 周）

- 实现 Gitea 数据采集器与存储。
- 实现 `PersonalBriefChain` 和基础日报渲染。
- 定时输出 Markdown 文件或发送到指定 Webhook。

### 阶段三：高级分析与协调（持续）

- 接入 `RiskClassifier` 和记忆模块。
- 增加团队协作分析链，提供主动提醒。
- 图表化趋势展示（可集成到简单 Dashboard）。

---

## 六、总结

本方案彻底改变了原有 `commit_guide` 的交互模式，将其转变为**自动化、非交互式**的提交说明生成工具，同时明确了服务端利用 `agent_engine` 实现管理日报的路径。这一设计既满足了员工端“懒人操作、详细生成”的需求，又通过本地生成详细消息降低了服务端 AI 的重复开销，实现了资源的合理分配。

后续开发可按阶段推进，优先交付客户端工具，快速验证可用性；再逐步完善服务端日报功能，最终形成完整的团队效能洞察体系。
